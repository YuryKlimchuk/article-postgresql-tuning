## Часть 4: Bitmap Scan и типы индексов — расширяем арсенал

В [предыдущей части](ссылка-на-часть-3) мы разобрали Index Scan и Index Only Scan — инструменты для точечных запросов. Но что делать, когда строк много (10-50% таблицы), а Seq Scan — перебор?

**В этой части:** Bitmap Heap Scan — компромисс между индексом и полным сканированием. А затем — обзор не-B-tree индексов: Hash для UUID, GIN для JSONB, BRIN для временных рядов.

---

## 5. Битовая карта (Bitmap Heap Scan)

### Двухфазный подход

Иногда строк много (~10-40% таблицы), но недостаточно для Seq Scan. Index Scan тоже плох: он будет прыгать по таблице в случайном порядке за каждой строкой.

PostgreSQL использует двухфазный трюк:

1. **Bitmap Index Scan** — проходит по индексу и строит в памяти битовую карту: «страница 5 — нужна (1), страница 6 — не нужна (0)»
2. **Bitmap Heap Scan** — идёт в таблицу и читает страницы **в физическом порядке** (по номеру страницы), а не в логическом порядке индекса

**Аналогия:** нужно собрать 50 книг из библиотеки. Index Scan — берёте список, бежите к первой книге на 3-й этаж, потом ко второй в подвал, потом к третьей на 1-й этаж. Bitmap Scan — сначала отмечаете на плане нужные полки, потом спокойно идёте от входа к выходу, собирая книги по пути. Физический порядок радикально снижает нагрузку на диск.

### Пример: чистая битовая карта

Вернёмся к эксперименту из раздела 2. Одна валюта (20% строк) на малой БД — это Bitmap Heap Scan:

**Малая БД (50K строк, 1 валюта = 10 000 строк / 20%):**

```
Bitmap Heap Scan on transactions
  (cost=113.00..1457.72 rows=9898 width=155)
  (actual time=0.654..3.568 rows=10000 loops=1)
  Recheck Cond: (currency_numeric = 643)
  Heap Blocks: exact=1221
  Buffers: shared hit=1239
  ->  Bitmap Index Scan on idx_transactions_currency_numeric
        (cost=0.00..110.53 rows=9898 width=0)
        (actual time=0.505..0.505 rows=10009 loops=1)
        Index Cond: (currency_numeric = 643)
        Buffers: shared hit=18
  Execution Time: 4.253 ms
```

`Heap Blocks: exact=1221` — все 1221 страницы таблицы прочитаны точно. Но даже при `exact` PostgreSQL делает `Recheck Cond`. Почему? Битовая карта показывает, какие **страницы** содержат нужные строки, но не гарантирует, что все строки на странице подходят под условие. PostgreSQL читает страницу целиком и фильтрует строки заново. Это плата за эффективность: вместо случайных чтений для каждой строки мы получаем последовательное чтение страниц, но с обязательной перепроверкой.

### Несколько условий: BitmapOr и BitmapAnd

Битовые карты можно комбинировать. PostgreSQL строит отдельную карту для каждого индекса и объединяет их:

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM transactions
WHERE card_id = 1 OR merchant_id = 5;
```

**Малая БД:**

```
Bitmap Heap Scan on transactions
  (cost=8.79..98.43 rows=25 width=155)
  (actual time=0.038..0.039 rows=25 loops=1)
  Recheck Cond: ((card_id = 1) OR (merchant_id = 5))
  Heap Blocks: exact=1
  Buffers: shared hit=4 read=1
  ->  BitmapOr
        ->  Bitmap Index Scan on idx_tx_card_amount_include
              Index Cond: (card_id = 1)
        ->  Bitmap Index Scan on idx_transactions_merchant_id
              Index Cond: (merchant_id = 5)
  Execution Time: 0.070 ms
```

**Большая БД:**

```
Bitmap Heap Scan on transactions
  (cost=9.04..92.63 rows=21 width=214)
  (actual time=0.048..0.049 rows=5 loops=1)
  Recheck Cond: ((card_id = 1) OR (merchant_id = 5))
  Heap Blocks: exact=1
  Buffers: shared hit=5 read=2
  ->  BitmapOr
        ->  Bitmap Index Scan on idx_tx_card_amount_include
        ->  Bitmap Index Scan on idx_transactions_merchant_id
  Execution Time: 0.086 ms
```

Две битовые карты → `BitmapOr` → одно чтение таблицы. Это позволяет использовать несколько индексов одновременно — то, чего не умеет обычный Index Scan.

### Главная ловушка: work_mem и Lossy Pages

Битовая карта строится в оперативной памяти. Её размер ограничен параметром `work_mem` (по умолчанию 4 МБ). Что если карта не влезает?

PostgreSQL не падает. Он переключается в режим **lossy** (потерянный): вместо бита на каждую строку запоминает бит на целую страницу. «Страница 42 — нужна, но какие именно строки в ней подходят — не помню».

Следствие: при чтении таблицы PostgreSQL читает страницу целиком и **перепроверяет каждую строку** (Recheck). Если страница большая, а подходящих строк мало — масса лишней работы.

**Эксперимент:** для демонстрации влияния `work_mem` мы принудительно заставим PostgreSQL использовать Bitmap Scan, отключив Seq Scan и Index Scan. В реальной жизни планировщик сам выберет оптимальный план. Но этот эксперимент показывает, что происходит, когда битовая карта не помещается в память:

```sql
SET enable_seqscan = OFF;
SET enable_indexscan = OFF;
SET work_mem = '64kB';

EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM transactions WHERE currency_numeric IN (643, 840);
```

```
Bitmap Heap Scan on transactions
  (cost=20746..239374 rows=1998667 width=214)
  (actual time=203.302..931.136 rows=2000000 loops=1)
  Recheck Cond: (currency_numeric = ANY ('{643,840}'::integer[]))
  Rows Removed by Index Recheck: 2994736
  Heap Blocks: exact=275 lossy=155976
  Buffers: shared hit=4 read=157820
  Execution Time: 981.754 ms
```

Кошмар: `lossy=155976` страниц из 156251 — почти всё «потеряно». `Rows Removed by Index Recheck: 2 994 736` — 3 миллиона строк прочитано и выброшено при перепроверке.

**Тот же запрос с work_mem = 4MB:**

```
Bitmap Heap Scan on transactions
  (cost=20746..231629 rows=1998667 width=214)
  (actual time=78.787..733.574 rows=2000000 loops=1)
  Recheck Cond: (currency_numeric = ANY ('{643,840}'::integer[]))
  Rows Removed by Index Recheck: 1897361
  Heap Blocks: exact=57431 lossy=98820
  Buffers: shared hit=1 read=157820
  Execution Time: 787.641 ms
```

`lossy` упал с 155K до 98K страниц, `Rows Removed` с 3M до 1.9M, время с 982ms до 788ms. Но сам принудительный Bitmap всё равно медленнее естественного Seq Scan (624ms — см. раздел 2).

Сравним:

| Метрика | work_mem = 64KB | work_mem = 4MB |
|---------|----------------|----------------|
| Heap Blocks: exact | 275 | 57 431 |
| Heap Blocks: lossy | **155 976** | 98 820 |
| Всего страниц | 156 251 | 156 251 |
| Rows Removed by Index Recheck | **2 994 736** | 1 897 361 |
| Execution Time | 981 ms | 788 ms |

Общее количество страниц одинаковое — разница только в точности битовой карты.

**Как настраивать work_mem:**

Для сессии:

```sql
SET work_mem = '64MB';
```

Глобально (`postgresql.conf`):

```
work_mem = 64MB
```

**Сколько ставить?** `work_mem` выделяется **на каждую операцию** сортировки или хеширования в запросе. Если у вас 100 одновременных соединений и в каждом сложный запрос с сортировкой — каждый получит свой `work_mem`.

Формула для оценки:

```
work_mem ≤ (total_RAM - shared_buffers) / (max_connections * 2)
```

Где `* 2` — запас на то, что в одном запросе может быть несколько операций сортировки/хеширования.

Например: 16 GB RAM, `shared_buffers = 4GB`, `max_connections = 100`:

```
work_mem ≤ (16GB - 4GB) / (100 * 2) = 60 MB
```

**Практическое правило:** если видите в `EXPLAIN` слова `lossy` и `Rows Removed by Index Recheck` — увеличивайте `work_mem`. Но не ставьте слишком большим: PostgreSQL может выйти за пределы RAM при высокой нагрузке. И главное — всегда проверяйте, не справится ли Seq Scan быстрее без всяких ухищрений.

**Итог:** Bitmap Heap Scan — компромисс между точечным Index Scan и полным Seq Scan. Идеален для выборки ~10-50% таблицы. Главная ловушка — `work_mem`: если битовая карта не влезает в память, производительность резко падает.

До сих пор мы использовали только B-tree индексы. В следующем разделе — краткий обзор других типов, и когда их стоит применять.

## 6. Типы индексов — краткий обзор

До сих пор мы использовали только B-tree — дефолтный и самый универсальный тип. Но B-tree не всегда оптимален. PostgreSQL поддерживает и другие структуры, каждая для своих задач:

| Тип | Когда использовать | Главная фишка |
|-----|-------------------|---------------|
| **B-tree** | 99% случаев. Равенство, диапазоны, сортировка. | Универсален, но фрагментируется при случайных вставках (UUID). |
| **Hash** | Только точное равенство (`=`). | Идеален для UUID. Не фрагментируется, занимает меньше места. |
| **GIN** | Полнотекстовый поиск, JSONB, массивы. | Строит обратный индекс. Поиск по содержимому составных типов. |
| **GiST** | Геометрия (PostGIS), нечёткий поиск. | Сложные операторы: пересечение, вхождение. |
| **BRIN** | Огромные таблицы с коррелированными данными (временные ряды, логи). | Хранит мин/макс для диапазонов страниц. Весит в сотни раз меньше B-tree. |

Разберём три самых полезных на практике.

### Hash-индекс: спасение для UUID

Если вы используете UUID как первичный ключ (популярно в микросервисах), B-tree-индекс будет постоянно фрагментироваться при вставке — UUID генерируются случайно.

Hash-индекс решает эту проблему: он вычисляет хэш от значения и хранит его в хэш-таблице.

```sql
CREATE INDEX idx_users_uuid_hash ON users USING hash(uuid);
```

До PostgreSQL 10 Hash-индексы были небезопасны для репликации (не писались в WAL). Сейчас это ограничение снято — Hash можно использовать в продакшене. Нюанс: Hash-индекс не поддерживает диапазоны (`>`, `<`) и сортировку (`ORDER BY`). Только точное равенство.

### GIN-индекс: поиск внутри JSONB и массивов

B-tree бесполезен для поиска внутри JSONB. GIN (Generalized Inverted Index) разбивает данные на лексемы и строит обратный индекс:

```sql
CREATE INDEX idx_users_names ON users USING gin(localized_names);
```

GIN-индексы могут быть большими, но незаменимы, когда нужно искать «иголку в стоге сена» внутри сложных структур. Подробный пример — в разделе 7.

### BRIN-индекс: магия для временных рядов

Если у вас таблица с миллиардами строк, где данные добавляются хронологически (например, `transaction_time`), B-tree-индекс будет весить гигабайты.

BRIN (Block Range Index) работает иначе: делит таблицу на диапазоны страниц и хранит для каждого только минимальное и максимальное значение. **Корреляция** означает, что физический порядок строк на диске совпадает с логическим порядком данных. Например, `transaction_time` растёт монотонно → строки на соседних страницах имеют близкие значения времени → BRIN эффективен. Если данные в случайном порядке (как `currency_numeric`), BRIN бесполезен — для каждой страницы мин/макс перекрывают весь диапазон.

```sql
CREATE INDEX idx_tx_time_brin ON transactions USING brin(transaction_time);
```

Сравнение размеров:

| БД | B-tree | BRIN | Разница |
|----|--------|------|---------|
| Малая (50K строк) | 1 112 KB | **24 KB** | ×46 |
| Большая (5M строк) | **107 MB** | **56 KB** | **×1 955** |

На 5 миллионах строк B-tree вырос до 107 МБ, а BRIN — всего 56 КБ. Почти **в 2 000 раз меньше**.

BRIN идеален для запросов вида `WHERE transaction_time > NOW() - INTERVAL '1 day'` — он быстро отбрасывает целые диапазоны страниц. Но бесполезен для точечного поиска (`WHERE transaction_time = '...'`).

**BRIN в деле:** сравним B-tree и BRIN на диапазонном запросе по большой БД (5M строк, ~400K за последние 30 дней):

**B-tree (idx_transactions_time, 107 MB):**

```
Bitmap Heap Scan on transactions
  (actual time=55.702..851.791 rows=398966 loops=1)
  Recheck Cond: (transaction_time > (now() - '30 days'::interval))
  Rows Removed by Index Recheck: 2880008
  Heap Blocks: exact=46073 lossy=99234
  Buffers: read=146400
  Execution Time: 864.232 ms
```

**BRIN (idx_tx_time_brin, 56 KB):**

```
Bitmap Heap Scan on transactions
  (actual time=4.577..1094.557 rows=398958 loops=1)
  Recheck Cond: (transaction_time > (now() - '30 days'::interval))
  Rows Removed by Index Recheck: 4601042
  Heap Blocks: lossy=156251
  Buffers: hit=6 read=156254
  Execution Time: 1107.326 ms
```

BRIN медленнее (1107 ms vs 864 ms) и вынужден перепроверить 4.6M строк — абсолютно все 156 251 страницы в режиме lossy. Но **индекс весит 56 КБ вместо 107 МБ** — в 2 000 раз меньше.

Обратите внимание: B-tree тоже показывает `lossy=99234` — это не особенность BRIN, а ограничение `work_mem` для битовой карты (раздел 5). Но B-tree всё равно точнее: у него `exact=46073` против `0` у BRIN — на первом этапе индекс отфильтровал часть страниц точно. BRIN же не может быть точным в принципе — он знает только мин/макс диапазона.

Если увеличить `work_mem`, BRIN ускорится — часть lossy-страниц станет exact (битовая карта будет точнее). Но даже с идеальным `work_mem` BRIN не догонит B-tree по скорости — его сила не в скорости, а в компактности.

Когда это оправдано? Если у вас таблица с миллиардами строк и ограниченный бюджет на RAM/диск для индексов — BRIN может быть единственным выходом. Для наших 5M строк B-tree всё ещё выигрывает.

### Как выбрать тип индекса

```
Нужен поиск по содержимому JSONB/массивов?
  → GIN

Ключ — UUID, нужен только поиск по равенству?
  → Hash

Данные добавляются хронологически (логи, события, транзакции)?
  → BRIN

Всё остальное:
  → B-tree (составной, если нужно)
```

**В нашей схеме:**
- `users.uuid` — кандидат на Hash-индекс (только равенство, не нужна сортировка)
- `users.localized_names` (JSONB) — кандидат на GIN (поиск внутри JSON)
- `transactions.transaction_time` — кандидат на BRIN (хронологические данные)

**Обратная сторона индексов:** каждый индекс замедляет `INSERT`, `UPDATE` и `DELETE` — PostgreSQL должен обновить не только таблицу, но и все индексы. В нашей схеме таблица `transactions` имеет 5 индексов (`card_id`, `merchant_id`, `transaction_time`, `currency_numeric` и первичный ключ). Каждая вставка обновляет все 5. На 50K строк это незаметно, на 5M и при высокой нагрузке — серьёзная плата за скорость чтения.

Как найти неиспользуемые индексы:

```sql
SELECT indexrelname, idx_scan, idx_tup_read
FROM pg_stat_user_indexes
WHERE schemaname = 'public' AND idx_scan = 0;
```

Если `idx_scan = 0` — индекс ни разу не использовался, его можно удалить. Чистый плюс к производительности записи. Не создавайте индексы «на всякий случай» — создавайте только под конкретный медленный запрос из `pg_stat_statements`.

**Итог:** B-tree покрывает 99% случаев, но знание альтернатив отличает Middle от Senior:

- **Hash** — идеален для UUID и точечного поиска по равенству, не поддерживает диапазоны и сортировку
- **GIN** — незаменим для JSONB, массивов и полнотекстового поиска
- **BRIN** — магия для временных рядов и логов, весит в тысячи раз меньше B-tree, но требует корреляции данных и может быть медленнее
- **GiST** — геометрия и сложные операторы (PostGIS)

Главное правило: не создавайте индексы «на всякий случай». Каждый индекс — это замедление `INSERT/UPDATE/DELETE` и дополнительная нагрузка на `VACUUM`. Создавайте индекс только под конкретный медленный запрос из `pg_stat_statements`.

## Сводная таблица типов сканирования

| Тип | Когда используется | Плюсы | Минусы |
|-----|-------------------|-------|--------|
| **Seq Scan** | Маленькая таблица ИЛИ большая выборка (>5-15%) | Простой, последовательное чтение | Медленный на больших таблицах |
| **Index Scan** | Точечный запрос или малая выборка | Быстрый для малых выборок | Случайные чтения в таблицу |
| **Index Only Scan** | Все колонки в индексе | Не ходит в таблицу | Требует покрывающего индекса |
| **Bitmap Heap Scan** | Средняя выборка, несколько условий | Эффективен для множества строк | Дополнительная память для битовой карты |
