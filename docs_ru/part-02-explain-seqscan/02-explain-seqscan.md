## 1. EXPLAIN ANALYZE — ваш новый лучший друг

Прежде чем разбираться с типами сканирования, нужно научиться смотреть «под капот» PostgreSQL. Для этого используется команда `EXPLAIN`.

### EXPLAIN vs EXPLAIN ANALYZE

Есть важное различие:

- **`EXPLAIN`** — показывает *план* выполнения запроса, который построил планировщик PostgreSQL. Запрос **не выполняется**, вы видите только оценку стоимости.
- **`EXPLAIN (ANALYZE)`** — **выполняет** запрос и показывает реальные метрики: сколько времени заняло, сколько строк было прочитано, сколько памяти использовано.

**Всегда используйте `EXPLAIN (ANALYZE, BUFFERS)`** для отладки `SELECT`-запросов. Для `UPDATE`/`DELETE` — оберните в транзакцию с `ROLLBACK`, чтобы не изменить данные. На проде даже `SELECT` без `LIMIT` может нагрузить диск — будьте осторожны.

### Ключевые метрики в выводе

Запустите первый тестовый запрос на **обеих** базах (не забудьте сначала обновить статистику):

```sql
-- Отключаем JIT-компиляцию, чтобы не загромождать вывод (JIT — тема отдельного разговора)
SET jit = OFF;

-- Отключаем параллельные воркеры, чтобы вывод совпадал с примерами в статье
SET max_parallel_workers_per_gather = 0;

-- Обновляем статистику (init-скрипты уже сделали ANALYZE, но на всякий случай)
ANALYZE transactions;

-- Тестовый запрос
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM transactions WHERE amount > 5000;
```

Вы увидите что-то вроде:

### Малая БД (50K транзакций)

```
 Seq Scan on transactions  (cost=0.00..1845.00 rows=25224 width=155)
                           (actual time=0.007..5.138 rows=25175 loops=1)
   Filter: (amount > '5000'::numeric)
   Rows Removed by Filter: 24825
   Buffers: shared hit=1220
 Planning Time: 0.056 ms
 Execution Time: 5.808 ms
```

### Большая БД (5M транзакций)

```
 Seq Scan on transactions  (cost=0.00..218750.00 rows=2533332 width=214)
                           (actual time=0.024..675.241 rows=2503844 loops=1)
   Filter: (amount > '5000'::numeric)
   Rows Removed by Filter: 2496156
   Buffers: shared hit=2702 read=153548
 Planning Time: 0.057 ms
 Execution Time: 735.447 ms
```

Разберём метрики на реальных цифрах из обеих БД:

| Метрика | Малая БД (`payment_small`) | Большая БД (`payment_large`) | Что означает |
|---------|---------------------------|------------------------------|--------------|
| **cost** | `0.00..1845.00` | `0.00..218750.00` | Оценка стоимости в условных единицах. Рост в 118 раз (218750 / 1845) отражает увеличение объёма данных. Cost — это оценка планировщика, не реальное время. |
| **actual time** | `0.007..5.138 ms` | `0.024..675.241 ms` | Реальное время: первое число — до первой строки, второе — общее. |
| **rows** | `25 175` | `2 503 844` | Сколько строк вернул узел. Ровно ×100 — данных больше, результат пропорционален. |
| **width** | `155` байт | `214` байт | Средний размер строки. На большой БД width больше, потому что `description` длиннее (`repeat(..., 5)` вместо `repeat(..., 3)`). TOAST на таких размерах ещё не включается — разберём в части 5. |
| **Buffers** | `hit=1220` | `hit=2702, read=153 548` | Страницы по 8 КБ. Малая БД: всё в кэше (1220 × 8 КБ ≈ 10 МБ). Большая БД: 2 702 в кэше + 153 548 с диска (~1.2 ГБ). Дефолтный `shared_buffers = 128 МБ` не вмещает таблицу в 5M строк — отсюда `read`. |
| **Rows Removed by Filter** | `24 825` | `2 496 156` | Сколько строк прочитано и отброшено. Seq Scan проверил каждую строку — половина не подошла. |
| **Planning Time** | `0.056 ms` | `0.057 ms` | Время построения плана. Практически одинаково! PostgreSQL не «тормозит» на больших данных — проблема в выполнении, а не в планировании. |
| **Execution Time** | `5.8 ms` | `735.4 ms` | Чистое время выполнения. ×127 при ×100 данных — почти линейный рост. |

Четыре вывода из этой таблицы:

1. **Разница ×127 по времени** при ×100 по данным — Seq Scan читает всё подряд, время растёт линейно с объёмом.
2. **`Rows Removed by Filter: 2 496 156`** — PostgreSQL прочитал 5 миллионов строк и половину выбросил. Seq Scan не умеет пропускать — он проверяет каждую.
3. **`Buffers: read=153 548`** (~1.2 ГБ с диска). Дефолтный `shared_buffers = 128 МБ` просто не вмещает таблицу — отсюда чтение с диска. Проверьте: `SHOW shared_buffers;`. На малой БД всё в кэше.
4. **`Planning Time` почти идентичен** — 0.056 vs 0.057 мс. PostgreSQL не «тупит» и не «думает дольше» на больших данных. Проблема всегда в выполнении (`Execution Time` растёт в 127 раз), а не в планировании.

## 2. Последовательное сканирование (Seq Scan)

### Что происходит

PostgreSQL читает **всю таблицу** страницу за страницей, проверяя каждую строку на соответствие условию `WHERE`. Это самый простой, но не всегда самый быстрый способ.

### Пример: нет индекса

Взглянем на уже знакомый запрос из первой части:

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM transactions WHERE amount > 5000;
```

На поле `amount` нет индекса → PostgreSQL вынужден читать всю таблицу. На малой БД это займёт миллисекунды, на большой — секунды.

### Seq Scan при наличии индекса — важный нюанс

Теперь запрос с индексом:

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM transactions WHERE currency_numeric = 643;
```

На поле `currency_numeric` **есть индекс** (мы создали его в `schema.sql`). Но планировщик выбирает разный план в зависимости от объёма данных:

**Малая БД (50K строк, 1 валюта = 10 000 строк / 20%):** планировщик выберет **Bitmap Heap Scan** — промежуточный вариант между Index Scan и Seq Scan (подробно разберём в части 4).

**Большая БД (5M строк, 1 валюта = 1 000 000 строк / 20%):** планировщик выберет **Seq Scan**. Один и тот же запрос, один и тот же процент — но на большом объёме даже 20% это миллион строк, и планировщик решает: «проще прочитать всё подряд».

Причина — в соотношении стоимости случайных и последовательных чтений (подробно — в следующем разделе):

**Правило:** Если планировщик оценивает, что нужно выбрать **> 5-15% строк**, он может выбрать Seq Scan. Точный порог зависит от `random_page_cost` (по умолчанию 4.0) и `effective_cache_size`. На SSD `random_page_cost` часто снижают до 1.5 — тогда порог становится **выше**, и Seq Scan включается позже (планировщик дольше держится за индекс).

**Проверьте на малой БД:** Там будет не Index Scan, а Bitmap Heap Scan (о нём — в части 4). Выполнится он мгновенно — даже «неоптимальный» план на малом объёме не проблема.

### Порог переключения: Bitmap → Seq Scan

На малой БД можно увидеть момент, когда планировщик меняет решение. Запустите три запроса с нарастающим числом валют:

```sql
-- 1 валюта = 10 000 строк (20% таблицы)
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM transactions WHERE currency_numeric = 643;

-- 2 валюты = 20 000 строк (40% таблицы)
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM transactions WHERE currency_numeric IN (643, 840);

-- 3 валюты = 30 000 строк (60% таблицы)
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM transactions WHERE currency_numeric IN (643, 840, 978);
```

Результат на малой БД:

| Валют | Строк | % таблицы | План | Execution Time |
|-------|-------|-----------|------|---------------|
| 1 | 10 000 | 20% | **Bitmap Heap Scan** | 2.050 ms |
| 2 | 20 000 | 40% | **Bitmap Heap Scan** | 3.380 ms |
| 3 | 30 000 | 60% | **Seq Scan** | 4.876 ms |

Порог пройден между 40% и 60% — добавили одну валюту, и планировщик передумал. Seq Scan при 60% всего на 1.5 ms медленнее Bitmap при 40% — выборка больше в 1.5 раза, а время выросло незначительно (спасибо последовательному чтению).

### На большой БД даже одна валюта — это миллион строк

```
Seq Scan on transactions  (cost=0.00..218750.00 rows=998833 width=214)
                          (actual time=0.023..466.996 rows=1000000 loops=1)
  Filter: (currency_numeric = 643)
  Rows Removed by Filter: 4000000
  Buffers: shared hit=2800 read=153450
  Execution Time: 492.563 ms
```

Сразу Seq Scan — 20% от 5 миллионов строк индексом не возьмёшь.

### Последовательный и случайный доступ — в чём разница

Чтобы понимать, почему планировщик принимает такие решения, нужно разобраться в двух способах чтения с диска:

**Последовательный доступ (sequential read)** — читаем страницы подряд, одну за другой. Диск (особенно HDD) делает это очень быстро: головка позиционируется один раз, дальше данные текут потоком. Именно так работает **Seq Scan**.

**Случайный доступ (random read)** — прыгаем между страницами, которые находятся в разных местах диска. Каждый прыжок — это перемещение головки (HDD) или поиск в таблице страниц (SSD). Именно так работает **Index Scan**: индекс указывает на строки, разбросанные по разным страницам таблицы.

PostgreSQL моделирует эту разницу через два параметра:

| Параметр | Значение по умолчанию | Что означает |
|----------|----------------------|--------------|
| `seq_page_cost` | 1.0 | Стоимость чтения одной страницы последовательно |
| `random_page_cost` | 4.0 | Стоимость чтения одной страницы в случайном порядке |

Соотношение 1:4 означает: *«один случайный прыжок стоит как 4 последовательных чтения»*. Это верно для HDD. Для SSD соотношение ближе к 1:1 или 1:1.5.

Теперь понятно, почему порог ~40-60%: если нужно прочитать половину таблицы случайными прыжками, это может стоить дороже, чем прочитать её всю последовательно. Планировщик считает стоимость обоих вариантов и выбирает дешевле.

**Аналогия из жизни:** представьте библиотеку. Последовательное чтение — это когда вы идёте вдоль полки и берёте книги подряд. Случайное — когда вам нужно найти 10 книг, разбросанных по разным этажам. Даже если каждая книга лёгкая, беготня по этажам займёт больше времени, чем просто прочитать всю полку.

### Эксперимент: меняем random_page_cost и наблюдаем смену плана

Самый наглядный способ понять влияние `random_page_cost` — поменять его на лету и увидеть, как PostgreSQL передумал:

```sql
-- Шаг 1. Три валюты: 60% таблицы. Смотрим план при random_page_cost = 4
SELECT current_setting('random_page_cost');  -- 4

EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM transactions WHERE currency_numeric IN (643, 840, 978);

-- Шаг 2. Делаем случайные чтения «дёшевыми» (как на SSD)
SET random_page_cost = 1.0;

-- Шаг 3. Тот же запрос — планировщик передумал!
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM transactions WHERE currency_numeric IN (643, 840, 978);

-- Шаг 4. Возвращаем на место
SET random_page_cost = 4.0;
```

### Результат на малой БД (50K строк, 3 валюты = 30 000 строк)

```
random_page_cost = 4.0:
  Seq Scan  (cost=0.00..1907.50) (actual time=0.009..4.110 rows=30000)
  Rows Removed by Filter: 20000
  Buffers: shared hit=1220
  Execution Time: 4.876 ms

random_page_cost = 1.0:
  Index Scan  (cost=0.29..1746.67) (actual time=0.037..3.421 rows=30000)
  Index Cond: (currency_numeric = ANY ('{643,840,978}'::integer[]))
  Buffers: shared hit=3687
  Execution Time: 4.159 ms
```

Поменяли один параметр — план переключился с Seq Scan на Index Scan.

### Результат на большой БД (5M строк, 3 валюты = 3 000 000 строк)

```
random_page_cost = 4.0:
  Seq Scan  (cost=0.00..225000.00) (actual time=0.022..581.101 rows=3000000)
  Rows Removed by Filter: 2000000
  Buffers: shared hit=2864 read=153386
  Execution Time: 653.590 ms

random_page_cost = 1.0:
  Index Scan  (cost=0.43..208274.96) (actual time=0.099..978.244 rows=3000000)
  Index Cond: (currency_numeric = ANY ('{643,840,978}'::integer[]))
  Buffers: shared hit=6 read=471105 written=6
  Execution Time: 1054.237 ms
```

И вот тут — **сюрприз**. Index Scan оказался **медленнее** Seq Scan (1054 ms vs 653 ms)! Почему?

- `Buffers: read=471 105` — Index Scan прочитал **в 3 раза больше страниц**, чем Seq Scan (153 386)
- Почему так много? Index Scan за каждой строкой идёт в таблицу (heap). При 3 миллионах строк — это 3 миллиона случайных заходов в разные страницы. Строки с `currency_numeric IN (643, 840, 978)` разбросаны по всей таблице, поэтому многие страницы читаются повторно — для разных валют на одной странице.
- Seq Scan читает каждую страницу **ровно один раз** — отсюда в 3 раза меньше I/O.
- `written=6` — страницы, «запачканные» при первом чтении (hint-bit updates: PostgreSQL проставляет служебные биты видимости). Для `SELECT` это нормально, объём копеечный.
- Планировщик при `random_page_cost = 1.0` думает, что случайные чтения дёшевы — и ошибается на таком объёме

**Мораль:** `random_page_cost` влияет на выбор плана, но не отменяет физику. Три миллиона случайных заходов в таблицу — это всегда дорого, даже на SSD.

**Важный нюанс про SSD:** `random_page_cost = 1.0–1.5` — правильная настройка для SSD, но она не делает случайный доступ бесплатным. Если выборка >20-30% таблицы, Seq Scan всё равно будет быстрее — потому что прочитать всё подряд эффективнее, чем прыгать, даже если прыжки дёшевы.

### Parallel Seq Scan

Если включить параллельные воркеры (`max_parallel_workers_per_gather = 2` — значение по умолчанию), PostgreSQL может запустить несколько процессов для параллельного чтения таблицы. Вы бы увидели что-то вроде `Workers Planned: 2` с `Parallel Seq Scan` — запрос ускоряется, но потребляет больше CPU.

> **Примечание:** в наших скриптах инициализации параллельные воркеры отключены (`SET max_parallel_workers_per_gather = 0`), поэтому в выводе выше вы видите простой `Seq Scan`, а не `Parallel Seq Scan`. Если включить (`max_parallel_workers_per_gather = 2`), PostgreSQL может запустить несколько воркеров для параллельного чтения таблицы — это ускоряет запрос, но потребляет больше CPU. В Docker с ограниченным CPU параллелизм может не активироваться.

Seq Scan — самый простой способ. В **Части 3** мы перейдём к индексам: как работает B-tree, почему Index Scan в сотни и тысячи раз быстрее Seq Scan на большом объёме, и как покрывающий индекс позволяет вообще не ходить в таблицу.
